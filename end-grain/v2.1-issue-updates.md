## Issue #19 — v2.1 Feature types

### Additions to the types

1. **`FeatureResult` needs per-kind variants.**
   The v2.1-design doc shows a single `FeatureResult` shape for all features.
   The walk-throughs show each feature kind needs different output fields:

   - **ComposeStrips result**: `{ featureId, status, statusReason?, panel }`
   - **Cut result**: `{ featureId, status, statusReason?, slices, offcuts, sliceProvenance }`
   - **Arrange result**: `{ featureId, status, statusReason?, panel, appliedEditCount, appliedEditSources, appliedSpacerCount, appliedSpacerSources }` <!-- spacer fields added after brick rewrite -->
   - **Preset result**: `{ featureId, status, statusReason?, expandedPlaceEdits }` (rotate/shift expansions) OR `{ featureId, status, statusReason?, expandedSpacers }` (for `spacerEveryRow` — see #30 notes)
   - **PlaceEdit result**: `{ featureId, status, statusReason? }` (no geometry output — consumed by Arrange)
   - **SpacerInsert result**: `{ featureId, status, statusReason? }` (also consumed by Arrange)

   Cleanest typing: `type FeatureResult = ComposeStripsResult | CutResult | ArrangeResult | ...`
   discriminated on `featureId`'s corresponding `Feature.kind`.

2. **`Preset.params` schema is per-preset.**
   Checkerboard uses `flipAlternate` → `params: {}`.
   Brick uses `flipAlternate` + SpacerInserts (NOT `shiftAlternate`, contrary to the original walk-through before its 2026-04-18 rewrite).
   Other presets will have more variants.

   Propose: discriminated union on `preset` name:
   ```ts
   type Preset =
     | { kind: 'preset'; id: string; arrangeId: string; preset: 'flipAlternate';    params: {};                              status: Status }
     | { kind: 'preset'; id: string; arrangeId: string; preset: 'rotateAlternate';  params: { degrees: 90 | 180 | 270 };     status: Status }
     | { kind: 'preset'; id: string; arrangeId: string; preset: 'mirrorAlternate';  params: {};                              status: Status }
     | { kind: 'preset'; id: string; arrangeId: string; preset: 'rotate4way';       params: {};                              status: Status }
     | { kind: 'preset'; id: string; arrangeId: string; preset: 'spacerEveryRow';   params: { species: Species; width: number }; status: Status }  // NEW — from brick rewrite
     | { kind: 'preset'; id: string; arrangeId: string; preset: 'shiftAlternate';   params: { shift: number };               status: Status };  // retained; may lack canonical home — see below
   ```
   <br>
   **`shiftAlternate` status**: originally proposed for brick. After the brick rewrite, no canonical pattern in the v2.1 list requires it. Keep as an advanced/uncommon preset for experimental use; flag in UI as "not part of the canonical pattern vocabulary."

3. **`PlaceEditOp` is well-specified by v2.1 already** — `reorder | rotate(90|180|270) | shift`. No changes needed.

4. **Arrange.appliedEditSources must track both preset-expanded and user-authored edits.**
   Each entry points back to the feature ID that produced the edit (either
   `preset-N` or `edit-N`). UI uses this for the "click tile to highlight
   originating feature" affordance at v2.13.

5. **Arrange needs separate spacer-bookkeeping fields** (added after brick rewrite):
   - `appliedSpacerCount: number`
   - `appliedSpacerSources: string[]` — each entry references a `SpacerInsert` feature id or a `preset` id (if from `spacerEveryRow`).
   Keeps spacer attribution distinct from placement-edit attribution so the timeline-panel UI can render them with different icons / colors.

### Rationale
- Walk-through § 5.2 (checkerboard) shows Preset's expanded edits are
  ephemeral — in pipeline output, not in timeline.
- Walk-through § 5.3 (checkerboard) shows Arrange's result needs
  `appliedEditCount` + `appliedEditSources` for the timeline-panel UI.
- Walk-through § 6 (brick, rewritten) shows `Preset.params` varies by preset name AND that Arrange's result needs parallel spacer bookkeeping for SpacerInsert attribution.

## Issue #20 — Pipeline.run() with cursor-slide algorithm

### Additions to the spec

1. **Execution order is topologically sorted, not linear over the timeline.**
   Arrange depends on FOUR categories of features targeting it (updated after brick rewrite):
   - Its upstream Cut (for slices + provenance).
   - Presets targeting it (for preset-expanded PlaceEdits and — via `spacerEveryRow` — preset-expanded Spacers).
   - PlaceEdits targeting it.
   - SpacerInsert features targeting it.

   `Pipeline.run()` must collect all four before executing Arrange.

   Simplest implementation: two-phase pass.
   - Phase 1: walk the timeline, index all Presets, PlaceEdits, and
     SpacerInsert features by `target.arrangeId`/`arrangeId`.
   - Phase 2: execute features in timeline order. When an Arrange executes,
     it consults the index built in phase 1.

2. **Preset expansion logic** (in execution phase):
   - For each Preset in the index, call its expansion function.
   - Expansion takes: (slices, sliceProvenance, params) → PlaceEdit[].
   - Each expanded PlaceEdit carries `source: preset.id` for trace/UI.
   - Expanded PlaceEdits include `target.contributingStripIds` populated
     from the upstream Cut's sliceProvenance.

3. **Trace order = execution order, not display order.**
   The trace array surfaces features in the order they actually ran.
   For checkerboard this means `compose → cut → preset → arrange`
   (preset before arrange), even though the user sees `[compose, cut,
   arrange, preset]` in the timeline panel.

4. **Arrange's output panel bbox may differ from its input's bbox** (updated after brick rewrite).
   Checkerboard: same bbox (flipAlternate is volume-preserving).
   Brick: SAME bbox (flipAlternate is volume-preserving here too — not shift as I originally thought).
   **But once SpacerInserts are applied to the brick, the bbox extends along the Z axis (cut normal direction).** 7 spacers × 5 mm = 35 mm z-extension.
   And `shiftAlternate` (if used) would extend along the X axis (perpendicular to cut normal).
   So: three directions of possible bbox change:
   - Cut: changes z-extent of each slice.
   - `shiftAlternate` (if used): extends x.
   - SpacerInsert (any arrangement): extends z.
   Cursor-slide and scene-camera code must not assume bbox is preserved.

5. **Tests that should exist at v2.2:**
   - Default timeline (2 strips, identity arrange) → flush bbox identity.
   - Mitred (rip=30, identity) → bbox matches input within epsilon.
   - Checkerboard (2 strips + flipAlternate preset) → 2×8 checker surface, bbox preserved.
   - Brick Step 2 (asymmetric compose + flipAlternate) → offset pattern visible, bbox preserved.
   - Brick Step 3 (asymmetric compose + flipAlternate + 7 SpacerInserts) → z-bbox extends by 7 × spacer_width.
   - Provenance on slices is populated correctly for rip=0 (all strips contribute).

### Rationale
- Walk-through § 5.1 (checkerboard) shows execution must be topological.
- Walk-through § 5.4 (checkerboard) shows expansion pulls from sliceProvenance.
- Walk-through § 5 (brick) shows shifts change the output bbox.

## Issue #30 — v2.12 Arrangement presets

### Additions to the spec

1. **Presets are persistent features, not immediate-mode generators.**
   Already in the v2.1 design doc. The walk-throughs confirm this is the
   right call: a Preset stays in the timeline and re-expands on every
   regen, so new slices (from a pitch change) automatically get the same
   treatment.

2. **Preset vocabulary and params** (updated after brick rewrite):
   - `flipAlternate` — params: `{}`
   - `rotateAlternate` — params: `{ degrees: 90 | 180 | 270 }`
   - `mirrorAlternate` — params: `{}`
   - `rotate4way` — params: `{}`
   - **`spacerEveryRow`** — params: `{ species, width }` — NEW, from brick rewrite. Expands into N−1 SpacerInsert features for an N-slice arrangement. Avoids 7-spacer-per-brick boilerplate.
   - `shiftAlternate` — params: `{ shift: number }` (mm, perpendicular to N). No canonical home in the current pattern list after the brick rewrite showed brick uses flipAlternate instead. Retain as advanced/uncommon preset.

3. **Composition rule with manual PlaceEdits.**
   When a Preset and a manual PlaceEdit target the same slice:
   - Proposed rule: **manual PlaceEdits always apply after preset-expanded
     ones** regardless of timeline order. Within each group, timeline order.
   - Rationale: user's direct action should always win over bulk action.
   - Alternative: strict timeline order — predictable but surprising for
     users who applied a preset, then did manual tweaks, then changed a
     cut parameter.

4. **"Alternate" semantics for odd slice counts.**
   `flipAlternate` on 8 slices: flip indices [1, 3, 5, 7].
   `flipAlternate` on 9 slices: flip indices [1, 3, 5, 7]? or [1, 3, 5, 7, 8]?
   Propose: indices where `i % 2 === 1`. Concrete, unambiguous.

5. **Edge handling for shift presets.**
   A `shiftAlternate(25)` pushes odd slices 25 mm perpendicular to N.
   Does Arrange's output:
   - Preserve the full extended bbox (half-bricks at edges)?
   - Trim to the input bbox (no half-bricks, but user loses material)?
   - Alternate shift direction (+25 and -25) to keep centered?

   Propose: **preserve the full bbox** by default. Real brick cutting
   boards have visible half-bricks and that's the maker aesthetic. A
   later "Trim" feature could post-process if a user wants clean edges.

6. **Shift direction convention.**
   `shift: 25` means +25 mm along the perpendicular-to-N axis, in the
   direction of positive X (or positive perpendicular component). Negative
   values shift the other way. Needs one line in the spec.

7. **Param validation.**
   - `rotateAlternate.degrees` must be 90 | 180 | 270 (type-enforced).
   - `shiftAlternate.shift` should be a finite number. Implementation
     should tolerate any finite value; extreme values produce warnings if
     they exceed reasonable bounds (e.g. > slice width).

### Rationale
- Walk-through § 6 (checkerboard) raised composition rule + "odd"
  semantics as open questions.
- Walk-through § 5 (brick) documents the shift preset's params + bbox
  behaviour + edge-handling policy.

## Brick rewrite — what changed (2026-04-18)

The original brick walk-through used a `shiftAlternate(25)` preset to create row offsets. A real-world reference (DIY Montreal) showed that canonical brick-pattern boards use a different recipe: asymmetric compose (half-brick at one end) + `flipAlternate` + SpacerInsert features for horizontal mortar. The walk-through and the design-doc Brick pattern card have been rewritten to use the canonical recipe.

Follow-on deltas to these notes:
- `spacerEveryRow` preset proposed (avoids boilerplate of N−1 SpacerInserts per arrangement).
- `shiftAlternate` may lack a canonical home; retain as experimental/advanced preset.
- Arrange.result needs `appliedSpacerCount` + `appliedSpacerSources` parallel fields.
- Pipeline dependency categories for Arrange expanded from 3 to 4 (added SpacerInsert).
- Two axes of bbox change: x (shiftAlternate) and z (Cut, SpacerInsert).

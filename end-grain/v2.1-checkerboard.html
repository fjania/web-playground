1. Recipe & target

From the v2.1 design doc's pattern card:

Compose(A,B,A,B,...) → Cut(rip=0) → Arrange + Preset(flipAlternate).

Target: an 8×8 checkerboard with 50 mm square cells. That means:

  • 8 strips alternating maple / walnut, each 50 mm wide (x-direction, total panel width = 400 mm).
  • Strip height 50 mm (y), strip length 400 mm (z).
  • Crosscut perpendicular to z (rip=0, bevel=90) at pitch 50 mm → 8 slices, each 400×50×50.
  • Arrangement: 8 slices stacked along the original z-axis. Without transforms every slice is identical, producing vertical stripes. flipAlternate rotates every odd-index slice 180° about its vertical axis so its first-species band becomes last — creating the checker.

To keep the JSON readable, the worked example uses a shorter default: 2 strips to start, then the user builds up. Final state = 8 strips.

Observations

  • Default timeline is 3 features long. No PlaceEdits, no Presets. The starting panel is the minimum viable (2 alternating strips) that still exercises every pipeline step.
  • Strip IDs start at strip-0. Monotonic counter owned by the ID allocator.
  • Every slice's contributingStripIds is the same at rip=0 — the cut spans all strips in x.   • The join tile shows vertical stripes, not a checker. That's correct for identity arrange — two strips × 8 slices means 2 visible bands repeated across the top.

Observations

  • Strip IDs continue the monotonic sequence. No gaps, no reuse. Six clicks added strip-2 through strip-7.
  • Cut slice count is derived, not stored. cut-0.pitch and compose-0.stripLength together determine slices.length = 8. If either changes, the pipeline re-derives.
  • Still no checker — identity arrange of 8 identical slices = the original panel. We need the preset.
  • Per-click regen. Every strip addition triggers a full pipeline run, not just an incremental update. The pipeline is fast enough (< 10 ms for this size) that this is fine.

    outputSummary: 'panel 400×50×400, 4 rotations applied', status: 'ok', durationMs: 2.1 }
]

Observations (this is the meaty step)

  • Preset is persistent, not immediate-mode. A single Preset feature sits in the timeline. Its expansion happens during pipeline regen, not at button-click time. If later a user changes cut pitch so 16 slices are produced, the preset re-expands to 8 PlaceEdits automatically.
  • Preset's expandedPlaceEdits live in the pipeline output, not in the timeline. They're ephemeral — regenerated each run. Only the Preset feature is persistent state.
  • Expanded edits carry the same target shape as manual PlaceEdits — including contributingStripIds. This lets the v2.14 resolver treat preset-expanded and user-authored edits uniformly.
  • Execution order ≠ timeline display order. The user sees [..., arrange-0, preset-0] in the timeline panel (preset was appended last). But the pipeline runs compose → cut → preset → arrange because arrange-0 depends on preset-0's expansion. Trace reflects the execution order; the UI displays user-addition order.
  • Arrange's result records appliedEditSources. This is what lets a UI click "where did this edit come from" on a specific slice — useful for the timeline-highlight affordance (v2.13).
  • flipAlternate's expansion logic is: for each odd-indexed slice, emit PlaceEdit(rotate 180). This is a pure function of slice count — it takes no parameters and is deterministic.

5. What this walk-through exposes

Writing out the concrete JSON at each step surfaced things that the design doc covered implicitly or not at all. These should land in #19 (v2.1 types) and #20 (pipeline) as concrete requirements.

5.1 Pipeline execution order is not timeline-display order

Arrange depends on any Preset targeting it, and presets need slice count from Cut. So pipeline execution is a topological sort of feature dependencies, not a linear walk of the timeline.

Implication for v2.2 (Pipeline.run()): the implementation must collect all features targeting each Arrange before Arrange executes. Simplest model: a two-phase approach — phase 1 walks the timeline to index all Presets and PlaceEdits by arrangeId; phase 2 executes features linearly, with Arrange consulting the index at execution time.

Implication for the trace: the trace is in execution order, not display order. The UI showing the timeline uses display order (user's addition sequence); the debug inspector's trace uses execution order. These are different views of the same state.

5.2 Preset expansion produces ephemeral PlaceEdits, not timeline entries

The Preset feature is persistent; its expandedPlaceEdits are transient pipeline output, regenerated each run. The design doc says "presets expand into PlaceEdits at regen" — this walk-through concretises that: the expanded edits live in PipelineOutput.results[presetId].expandedPlaceEdits, never in AppState.timeline.

Implication for v2.1 types: FeatureResult discriminated union needs a variant for Preset with expandedPlaceEdits: PlaceEdit[]. The v2.1-design doc's FeatureResult shape doesn't currently include this — needs added.

5.3 Arrange's result must surface "which edits applied" for downstream UI

The checkerboard example shows 4 rotations applied. For the v2.13 timeline panel to highlight-on-click, the UI needs to know which slices were transformed by which feature. Hence appliedEditSources on the Arrange result — a parallel array to the applied edits, pointing back to the feature ID that generated each.

Implication: Arrange's result isn't just a panel — it carries bookkeeping. Needs to be a { panel, appliedEditCount, appliedEditSources } shape in the FeatureResult for Arrange.

5.4 Cut output's sliceProvenance is the canonical target-reference source

When Preset expands at regen, it needs to populate each expanded PlaceEdit's target.contributingStripIds. That metadata comes straight from cut-0's result. So the expansion logic is:

for sliceIdx in oddIndices:
  target = {
    arrangeId: preset.arrangeId,
    sliceIdx,
    contributingStripIds: cutResult.sliceProvenance[sliceIdx].contributingStripIds
  }
  emit PlaceEdit(target, rotate(180), source: preset.id)

Implication for v2.2: Pipeline.run() needs to pass the appropriate Cut result into each Preset's expansion function.

5.5 Strip ID monotonicity under sequential adds is trivially verified by this example

6 clicks → strip-2 through strip-7. No gaps, no reuse. The ID allocator is just a monotonic counter; no more sophistication needed unless and until deletion-then-add creates a motivation to prove "never reused" holds.

5.6 For rip=0, provenance discriminates nothing

Every slice's contributingStripIds is identical in checkerboard (all 8 strips). So at v2.14, provenance-first resolution adds no value here — sliceIdx carries the load. That's fine: checkerboard was never the test case for provenance resolution. The interesting cases are mitred cuts where different slices have different contributing strips. This walk-through confirms checkerboard doesn't exercise that path (and doesn't need to).

5.7 The Feature vocabulary is sufficient for checkerboard

Final timeline has 4 features: ComposeStrips, Cut, Arrange, Preset. No PlaceEdit, no SpacerInsert. Checkerboard's full expressiveness is covered by the base vocabulary plus the Preset feature kind. Good sanity check that Preset isn't redundant with PlaceEdit.

6. Open questions after this exercise

  • What happens when Preset + PlaceEdit target the same slice? E.g. user applies flipAlternate, then manually flips slice 1 back. The Preset wants slice 1 flipped; the manual PlaceEdit wants it un-flipped. Both are rotate(180), so naïvely they stack to rotate(360) = identity. That happens to produce the right result in this case, but it relies on cancellation being well-defined. Other cases (preset flipAlternate + user shift on slice 1) compose differently. We need an explicit ordering rule. Proposed rule for v2.12: manual PlaceEdits always apply after preset-expanded ones; within each group, by timeline order.
  • Does flipAlternate interpret "odd" as slice indices 1, 3, 5… or as the slices in the second half of the alternation pattern? For 8 slices, both give the same result. For 9 slices, they differ. Spec needs to pin this.
  • What does cursor-slide do when a slice is rotated 180° but the slice is symmetric along the rotation axis? For rip=0 checkerboard, each slice is symmetric — rotation shouldn't change its geometry. Only the strip labelling differs (maple becomes walnut-position, walnut becomes maple-position). Cursor-slide measures thickness-along-N, which is invariant under this rotation. So the arranged panel's bounding box is identical before and after the preset — only the species labelling of the surface changes. Worth confirming the algorithm handles symmetric slices cleanly.
  • The Preset.params field is shown empty for flipAlternate. Other presets (shiftAlternate with a delta) will populate it. Worth sketching a second walk-through page for brick to surface how params is typed and validated.

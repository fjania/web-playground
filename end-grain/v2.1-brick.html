1. Recipe & target

From the v2.1 design doc's pattern card:

Compose(A,B,A,B,...) → Cut(rip=0) → Arrange + Preset(shiftAlternate, ½ brick).

Target: a running-bond brick pattern with 50×50 mm cells and half-cell row offsets. Same compose + cut parameters as the checkerboard walk-through; the only difference is the preset kind and its param.

  • 8 strips alternating maple / walnut, each 50 mm wide. Panel 400×50×400.
  • Crosscut perpendicular to z at pitch 50 mm → 8 slices, each 400×50×50.
  • Arrangement: 8 slices stacked along z; shiftAlternate({ shift: 25 }) translates every odd-index slice by 25 mm along the X axis (perpendicular to the cut normal). Half of a 50 mm brick = 25 mm.

Because this walk-through shares its starting point with the checkerboard one, Steps 0 and 1 are condensed — same panel, same cut, same identity arrange. The real work is in Step 2.

Observations

  • Preset.params carries data now. flipAlternate had empty params. shiftAlternate has { shift: 25 }. Confirms Preset's type must be a discriminated union on preset name, with per-preset params shapes.
  • Output panel bbox is 425×50×400, not 400×50×400. The +X extension comes from odd slices overhanging. This is new: flipAlternate preserved the bbox, shiftAlternate extends it.
  • Half-bricks at both X edges. At x=0..25 mm: only even slices have material. At x=400..425 mm: only odd slices. Real brick boards accept this aesthetic (or trim the overhangs after glue-up). Our model preserves the overhangs by default.
  • The cut tile is literally identical to Step 1 / to checkerboard Step 2's cut tile. The preset changes nothing upstream of Arrange. This is the whole point of a feature timeline — distinct authorial intents compose cleanly.
  • Expanded PlaceEdits share the same target shape with manual PlaceEdits, so the resolver at v2.14 treats preset-generated and user-authored edits uniformly — whether the op is rotate or shift.
  • Pipeline execution order is the same topological sort as checkerboard: compose → cut → preset → arrange. Preset runs before Arrange because Arrange depends on its expansion.

5. Invariants of the shift operation

The shift op on a slice preserves certain properties and changes others. Making these explicit tells the pipeline what to test for and tells the visualizations what to make visible.

Propertyshift(25)rotate(180) (from checker for comparison)
Slice count preserved preserved
Per-slice band composition (species × x-interval) preserved preserved (just mirrored in x)
Per-slice band ordering along x preserved reversed
Slice position along stack axis (Z) preserved (cursor-slide still abuts along N) preserved
Slice position along perpendicular axis (X) shifted by +25 mm on odd slices preserved
Output panel bbox extended from 400 to 425 mm in X preserved
Volume of material preserved preserved

The join tile above shows this: every slice still has its full M W M W M W M W band composition (eight 1-unit cells), eight slices are still stacked flush along Z (horizontal mortar lines at each slice boundary), and the only thing that changed is that odd slices live at x ∈ [0.5, 8.5] instead of x ∈ [0, 8]. The dashed orange box shows the original 400×400 bbox so the extension is obvious.

6. What this walk-through newly exposes

Everything the checkerboard walk-through exposed still applies. Brick adds the following:

6.1 Preset.params is per-preset-kind

Concrete instances: flipAlternate takes no params. shiftAlternate takes { shift: number } (mm). Future additions per the v2.1 design:

  • rotateAlternate: { degrees: 90 | 180 | 270 }
  • mirrorAlternate: no params
  • rotate4way: no params

Type-level: Preset must be a discriminated union keyed on preset name. Written up as a concrete proposal in issue-update notes, §#19.

6.2 Arrange's output bbox is not always the input bbox

The checkerboard preset left the bbox untouched. The brick preset extends it by shift mm in X. This matters for:

  • Tests: v2.2's pipeline test suite must assert bbox extension for shift presets and bbox invariance for rotate presets — two separate assertions.
  • Downstream passes: if a second Cut feature operates on the arrange-0 output, it sees a wider panel than the original. Provenance and slice counts cascade accordingly.
  • Scene rendering: the tile camera needs to fit the actual output bbox, not the input bbox. Simple in practice — the render loop reads arrange.panel.bbox, not compose.panel.bbox.

6.3 Edge-handling is a user-facing decision

The output has half-bricks at both X edges. Three plausible behaviours:

  1. Preserve the full extended bbox (this walk-through's default — realistic to how makers actually build brick boards, trimming after if they want).
  2. Trim to the original input bbox (clean edges but material loss). Would be a separate Trim feature.
  3. Symmetric shift: alternate +shift/2 and −shift/2 instead of 0/+shift so the pattern stays centered. Keeps bbox asymmetric issues but avoids one-sided overhang.

Recommendation: go with option 1 for v2.12 (simplest, most realistic, and 2 + 3 can be built later without breaking 1).

6.4 Direction convention for shift

Positive shift goes in +X direction (perpendicular to N, toward positive X of the workpiece). Negative shifts the other way. Needs one sentence in the v2.12 spec. Same convention applies to shiftAlternate and to manual PlaceEdit(shift, delta).

6.5 Shift preset's expansion must receive its params

The preset-expansion signature has to thread the preset's params through to the expansion function:

expand(preset, slices, sliceProvenance): PlaceEdit[]
  → for 'flipAlternate': { slices, provenance } → rotate(180) on odd indices
  → for 'shiftAlternate': { slices, provenance, params.shift } → shift(params.shift) on odd indices

Issue #20's spec should surface this signature. It's already noted in the issue-updates notes.

6.6 Preset + manual composition gets more interesting with shifts

In checkerboard, Preset(flipAlternate) + manual PlaceEdit(rotate 180) on the same slice happen to cancel (since rotate is its own inverse). In brick, Preset(shiftAlternate, 25) + manual PlaceEdit(shift, 10) on the same slice compose linearly: the slice ends up shifted by 25 + 10 = 35. Or does it? Two possibilities:

  1. Additive: compose. Slice ends up shifted by 35 mm.
  2. Override: manual wins. Slice ends up shifted by 10 mm only.

Preferred rule (stated in the issue-updates notes): manual always wins over preset on the same slice, regardless of op kind. So the manual PlaceEdit replaces the preset's contribution for that slice. This is consistent, predictable, and makes the checkerboard workflow 8.4 in v2.1-design.html ("manual override of a preset") work uniformly across op types.

7. Open questions after this exercise

  • What if shift exceeds strip width? e.g. shift = 60 on 50 mm strips. The band offsets cross more than one band, producing a pattern very different from "brick." Not broken, just surprising. Does the UI clamp, warn, or allow freely? Proposed: allow freely, surface the current shift value prominently in the controls.
  • What if shift is negative? Same geometry mirrored. Works by the convention above (+X vs −X). No special handling needed, but worth a sentence of spec.
  • What if strips aren't all the same width? The "half-brick" heuristic only makes sense when bricks have a uniform width. With a mixed-width panel (the "chaos" pattern's starting state), shiftAlternate(25) still produces a shifted pattern but not a brick. The preset doesn't care — it's a pure translation op — but the name shiftAlternate is only meaningful when bricks align.
  • Should the shift amount be expressed in mm or as a fraction of strip width? Currently absolute mm. The pattern card says "½ brick" suggesting a fraction. If we store absolute mm, the preset stops making sense when a user later changes strip width. If we store a fraction, the preset auto-rescales. Recommendation: store mm, with a UI affordance to set it as "½ current strip width" at apply time. Predictable at regen, convenient at creation.

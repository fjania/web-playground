End Grain v2.1 — Design Review

Revision of the v2 feature-timeline design, updated to cover the real-world pattern vocabulary of end-grain cutting boards.
Supersedes: v2-design.html
Parent issue: #18
Architectural doc: #6
Current app: 3d.html

What changed from v2 → v2.1

  • Explicit pattern vocabulary. The app's purpose is to let makers experiment with named and freeform end-grain patterns. This revision enumerates the patterns it must support and names the ones it deliberately doesn't.
  • PlaceEdit ops extended from {reorder, flip, shift} to {reorder, rotate(90|180|270), shift}. Unblocks basket weave, pinwheel, chevron. flip becomes UI shorthand for rotate(180).
  • New SpacerInsert feature for accent-strip insertion at arrange-time. Unblocks Celtic knot and similar decorative patterns.
  • Arrange no longer owns placements. PlaceEdits live only in the timeline. Fixes a spec inconsistency in the original v2 plan.
  • Status field added from day one in v2.1 types (not retrofitted at v2.8).
  • Richer PlaceEdit target from day one — contributingStripIds captured but unused until v2.14. Removes the v2.9 → v2.14 migration.
  • Presets pre-committed as persistent features (not immediate-mode batches). Matches "every intent is a feature" and makes upstream changes replay correctly.
  • Axis-lock math specified for transverse drag (v2.11) — projection into {cutNormal, perpInPlane} basis.
  • Two new steps appended: v2.16 (rotate 90/270 verbs + basket-weave/pinwheel verification), v2.17 (SpacerInsert implementation + Celtic knot verification).
  • Explicit out-of-scope list — tapered strips (compass rose), non-uniform slice thickness, recursive strips, post-assembly inlay/shaping.

1. Elevator summary

The app lets makers design end-grain cutting boards virtually before committing them to the shop. It must support the named pattern vocabulary the woodworking community uses (checkerboard, brick, 3D cube, herringbone, chevron, basket weave, pinwheel, chaos, Celtic knot, quilt, pixel, diamond, ripple) and must also accommodate the experimental freeform patterns makers push into when they're not working from a recipe.

Every user intent — compose strips, cut, arrange, place a slice, insert an accent — becomes an immutable Feature in an ordered timeline. The final 3D result is the composition of every feature, evaluated top-down, regenerated on any upstream change. Each feature stores references to entities from earlier features; references are re-evaluated on every regen. A reference that can't be resolved marks its feature warning (found a close match and computed) or error (can't compute, feature skipped).

The user does not have to redo downstream work after an upstream change unless a reference genuinely breaks. When it does, the exact affected feature is surfaced with a badge and one-click reselect or delete.

Alongside the timeline, slices are manipulated directly in 3D: drag to reorder, double-click to rotate 180° (flip), right-click to open rotate options for 90°/270°, drag transversely to shift. Each manipulation writes a PlaceEdit into the timeline. Presets (flipAlternate, rotateAlternate, etc.) are persistent features that expand into PlaceEdits at regen, so they keep applying to new slices when upstream parameters change.

2. Design principles

  • Every user intent is a Feature. Parameter changes and manual edits both land in the same ordered list.
  • Pattern vocabulary is explicit. The app names what it supports and names what it doesn't. Freeform experimentation lives inside the supported vocabulary, not outside it.
  • References are re-evaluated each regen, not stored as opaque IDs. Each resolver is a small function.
  • Two failure modes only. warning (approximate resolution succeeded) or error (reference lost, feature skipped). Mirrors Fusion's yellow/red.
  • Stable IDs, allocated at creation, never reused after deletion.
  • Cursor-slide arrangement (the craftsman's algorithm) replaces AABB-based layout. Identity becomes flush reassembly regardless of cut angle.
  • Presets are persistent. A preset is a feature that re-generates its batch of PlaceEdits on each regen, so slice-count changes don't leave the new slices un-patterned.
  • No auto-heal. Heuristic "nearest match" repairs are confusing bug sources. Clear warning + one-click reselect.
  • No query language. Reference vocabulary is small enough to hand-write resolvers.
  • Greenfield, not migration. New files under src/v2/; current app untouched during the build; final step archives old files.

3. Supported patterns new

These are the patterns the app must be able to represent. Each is expressed as a short feature sequence. The visual tiles are stylised SVG — they show the geometric logic of each pattern, not the actual renderer output.

Checkerboard walk-through →

single pass · classic
Compose(A,B,A,B,...) → Cut(rip=0) → Arrange + Preset(flipAlternate).

Brick

single pass · shift
Compose(A,B,A,B,...) → Cut(rip=0) → Arrange + Preset(shiftAlternate, ½ brick).

3D cube / isometric

multi-pass · mitred
Three-species compose → Cut(rip=0) → Arrange → Cut(rip=30°, bevel=60°) → Arrange. Strip widths tuned for rhombus angles.

Herringbone

multi-pass · mitred
Striped compose → Cut(rip=45°) → Arrange + Preset(flipAlternate).

Chevron

multi-pass · mitred
Striped compose → Cut(rip=45°) → Arrange + Preset(mirrorAlternate).

Basket weave

multi-pass · rotate(90)
ABA sandwich compose → Cut(rip=0) → Arrange + Preset(rotateAlternate, 90°).

Pinwheel

multi-pass · 4-way rotate
Sub-block compose → Cut(rip=0) → Arrange + Preset(rotate4way).

Quilt

multi-pass · block-based
Multi-species compose → Cut(rip=0) → Arrange + free PlaceEdits / preset combinations for block motifs.

Pixel / mosaic

single pass · image-like
Compose with per-strip species sequence designed so that after cut + identity arrange, the target image emerges.

Chaos

single pass · non-uniform widths
Compose with varying per-strip widths and many species → Cut → Arrange + free PlaceEdits (reorder to taste).

Diamond

multi-pass · mitred
Striped compose → Cut(rip=45°) → Arrange + Preset(rotate4way).

Ripple

single pass · progressive widths
Compose with progressively-varying strip widths → Cut → Arrange. Staircase approximation; no actual curves.

Celtic knot v2.17

single pass · accent strips
Compose with thin accent strips → Cut(rip=0) → Arrange + Preset(shiftAlternate) + SpacerInserts between slices.

Freeform experimentation is expected — makers combine these recipes, mix presets with manual PlaceEdits, layer multiple cut-arrange passes, and compose sub-panels through the timeline. The feature vocabulary is designed to let that happen inside the model rather than against it.

4. Deliberately out of scope new

These patterns and techniques exist in the real-world maker vocabulary but are not supported by the v2.1 model. They're named here so the boundary is explicit — and so post-v2 work can pick them up without re-discovering the constraint.

Technique / patternReason it's outPossible future path
Compass rose, radial patterns Requires tapered (non-rectangular) strips. Strip model is {species, width} with implicit parallel edges. Post-v2 TaperedStripDef + domain support for parallelogram/trapezoid strip geometry.
Multi-species within a single strip (laminated strips) Needed for some basket-weave and 3D-cube variants. Strips are atomic in v2.1. Post-v2 RipStrips feature — output of a prior pipeline becomes the strip input of a later one.
Non-uniform slice thickness in one cut pass Cut has a single pitch. Perfect chaos and exact ripple curves need varying thickness. Multiple sequential single-slice cuts get you there slowly. A VaryingCut feature could replace pitch with a per-slice thickness array.
Post-assembly shaping (paddle shape, curved edges) The model produces rectangular panels. Shaping happens at the bandsaw / router after glue-up. Out-of-scope long-term; real workflow separates design from shaping.
Inlays (routed pockets + fill) Combines end-grain base with a second additive step not representable as strip composition. Out-of-scope long-term.
3D rotation exposing side grain Deliberately excluded — breaks the end-grain face-up invariant. All PlaceEdit rotations are about the vertical axis. Not planned.

6. Logical concepts

Feature

One user intent with persistent parameters and a status. Immutable once created; edited via parameter updates. Kinds: ComposeStrips, Cut, Arrange, PlaceEdit, Preset, SpacerInsert.

Timeline

Ordered list of features. The final 3D result is the composition of every feature, evaluated top-down. The authoring surface.

Reference

How one feature points at an entity produced by an earlier feature. Re-evaluated each regen. Examples: { stripId }, { arrangeId, sliceIdx, contributingStripIds }.

Resolver

Small function that converts a reference to a live entity at regen time. Returns ok, warning+match, or error.

Status

Per-feature 'ok' | 'warning' | 'error'. Present on every feature from day one. Rendered as tile badges + timeline-panel indicators.

Provenance

A slice's contributingStripIds — the strip IDs whose geometry feeds it. Captured by every PlaceEdit from v2.1 onward; consumed by the robust resolver at v2.14.

Cursor-slide

The arrangement algorithm. For each slice in order, measure its thickness along N, place its min-along-N face at the running cursor, advance. No pitch, no AABB, no spacing formula. Flush by construction regardless of cut angle.

PlaceEdit updated

A feature that applies a single manipulation — reorder, rotate(90|180|270), or shift — to one slice in one arrangement. Stored as its own timeline entry; stackable with other PlaceEdits on the same slice.

Preset updated

A persistent feature that generates a batch of PlaceEdits at regen. Re-evaluates against the current slice list, so new slices (e.g. after a pitch change) get the same treatment. Individual PlaceEdits can be manually overridden.

SpacerInsert new

A feature that inserts a thin accent strip between two slices in an arrangement. Carries species, width, and a target reference (arrangeId + insertion-position reference). Canonical use: Celtic knot. Reference failure modes mirror PlaceEdit's.

Tile

One scissored viewport on the shared Three.js canvas. Tile has a root Group, overlays, pickable groups, and a rect. Rendering iterates tiles; only the current tile's geometry is visible during its render pass.

7.2 — Features (discriminated union) updated

type Feature =
  | ComposeStrips
  | Cut
  | Arrange
  | PlaceEdit
  | Preset
  | SpacerInsert;

interface ComposeStrips {
  kind: 'composeStrips';
  id: 'compose-0';                // exactly one per design, stable id
  strips: StripDef[];             // ordered left-to-right
  stripHeight: number;            // uniform Y
  stripLength: number;            // uniform Z
  status: Status;                 // from v2.1
}

interface Cut {
  kind: 'cut';
  id: string;                      // 'cut-0', 'cut-1', ...
  rip: number;                     // -90..90°
  bevel: number;                   // 45..90°
  pitch: number;                   // mm
  showOffcuts: boolean;
  status: Status;
}

interface Arrange {
  kind: 'arrange';
  id: string;                      // 'arrange-0', 'arrange-1', ...
  layout: 'cursor-slide';          // only one algorithm for now
  status: Status;
  // NB: placements do NOT live here. PlaceEdits live in the timeline
  // and are discovered by arrangeId.
}

interface PlaceEdit {
  kind: 'placeEdit';
  id: string;                      // 'edit-0', 'edit-1', ...
  target: PlaceEditTarget;
  op: PlaceEditOp;
  status: Status;
  statusReason?: string;
}

interface Preset {
  kind: 'preset';
  id: string;                      // 'preset-0', ...
  arrangeId: string;               // which Arrange it operates on
  preset:
    | 'flipAlternate'
    | 'rotateAlternate'            // 90° alternate, for basket weave
    | 'rotate4way'                 // quadrant rotations, for pinwheel/diamond
    | 'mirrorAlternate'            // chevron
    | 'shiftAlternate';
  params?: { shift?: number };     // e.g. shiftAlternate's delta in mm
  status: Status;
}

interface SpacerInsert {
  kind: 'spacerInsert';
  id: string;                      // 'spacer-0', ...
  arrangeId: string;
  afterSliceIdx: number;           // inserts accent after this slice
  contributingStripIds?: string[]; // provenance of the slice before
  species: Species;
  width: number;                   // mm, along the stack axis
  status: Status;
  statusReason?: string;
}

7.3 — References updated

PlaceEdit targets carry both the positional reference and the provenance set from v2.1. The resolver uses positional only until v2.14, where it upgrades to provenance-first:

interface PlaceEditTarget {
  arrangeId: string;
  sliceIdx: number;                         // always stored; resolver fallback
  contributingStripIds?: string[];          // captured at edit time; consumed at v2.14
}

Resolver order at v2.14:

  1. Find slice whose contributingStripIds exactly matches stored set → apply, status ok.
  2. Fall back to sliceIdx positional match if still in range → apply, status warning.
  3. Otherwise → status error, edit skipped.

7.4 — PlaceEdit ops updated

type PlaceEditOp =
  | { kind: 'reorder'; newIdx: number }
  | { kind: 'rotate'; degrees: 90 | 180 | 270 }   // rotation about slice's vertical axis
  | { kind: 'shift'; delta: number };              // mm, perpendicular to cut normal
Note on "flip". The UI term flip is preserved in user-facing copy and corresponds to rotate(180). Double-click on a slice creates a rotate(180) PlaceEdit; double-click again removes it.

7.5 — Rotation semantics

All rotations are about the slice's vertical axis (perpendicular to the cut face, i.e. the axis that keeps end grain facing up). 90° and 270° swap the slice's horizontal dimensions — used for basket weave and pinwheel. 180° mirrors without affecting end-grain orientation — the common "flip" op.

Rotation is applied before cursor-slide. Because cursor-slide measures each slice's thickness along the cut normal dynamically, the rotated slice's new thickness is consumed correctly — no gaps or overlaps regardless of cut angle or rotation amount.

interface StripDef {
  stripId: string;                          // stable; never reused
  species: Species;
  width: number;                            // mm
}

type Species = 'maple' | 'walnut' | 'cherry' | 'padauk' | 'purpleheart';

interface FeatureResult {
  featureId: string;
  status: Status;
  statusReason?: string;
  panel?: Panel;                              // ComposeStrips, Arrange
  slices?: Panel[];                           // Cut

interface PipelineStep {
  featureId: string;
  kind: Feature['kind'];

}

9. Workflows

9.1 — Build a starting panel

  1. Open /end-grain/3d-v2.html. The default timeline contains ComposeStrips + Cut + Arrange.
  2. Set per-species widths in the catalog. Click + next to any species to append a strip. Panel widens.
  3. Drag strips along X in the panel tile to reorder. Release commits.
  4. Right-click a strip to remove it.
  5. Adjust height/length sliders for the whole panel.

9.2 — Build a checkerboard

  1. Default timeline already has 16 alternating maple/walnut strips.
  2. Open the Arrange controls. Click the flipAlternate preset button.
  3. A new Preset(flipAlternate, arrangeId=arrange-0) feature appears in the timeline.
  4. The join tile renders the classic checker. Internally the preset expanded into N/2 PlaceEdit(rotate 180) entries on odd slices at regen.
  5. Change pitch to halve the slice count — the preset re-expands against the new slice list and the checker still covers everything.

9.3 — Mitered rip without gaps

  1. Set rip to 31° in the Cut controls.
  2. Leave Arrange at default (cursor-slide identity).
  3. Observe the join tile: slices reassemble flush — no gaps, no overlaps. The joined panel's bounding box equals the starting panel's.
  4. This works because cursor-slide reads each slice's thickness along N from the geometry and places successive slices meeting face-to-face, rather than guessing from AABB centers.

9.4 — Manual override of a preset

  1. Apply the flipAlternate preset. The timeline grows by one Preset feature.
  2. At regen, the Preset expands into PlaceEdit(rotate 180) for each odd slice.
  3. Decide slice 2 should not be flipped. Right-click → "exclude from preset" (UI detail), which emits a PlaceEdit(rotate 180) on slice 2 as an explicit override that cancels the preset's edit for that target.
  4. Drag slice 5 transversely by 10mm. A new PlaceEdit(shift) is appended.
  5. All edits are explicit in the timeline; upstream changes replay the preset + the override correctly.

9.5 — Basket weave via rotate(90) v2.16

  1. Compose ABA sandwich strips (maple-walnut-maple repeating).
  2. Cut at rip=0° with pitch matching the panel height so slices end up square.
  3. Apply rotateAlternate preset. Every odd slice rotates 90° about its vertical axis.
  4. The join tile shows the interlocked basket weave.

9.6 — Celtic knot via SpacerInsert v2.17

  1. Compose a panel with thin padauk accent strips sandwiched between maple/walnut.
  2. Cut at rip=0°.
  3. Apply shiftAlternate preset with shift equal to the accent-strip width so accent lines align across slices.
  4. In the join tile, insert SpacerInserts of padauk between adjacent slices where the knot calls for a crossing.
  5. The inserted spacers carry provenance; changing the accent strip's species upstream updates the spacers' target and the knot stays coherent.

9.7 — Upstream change, downstream survival

  1. Existing design: composed panel + Cut(rip=0°) + Arrange + Preset(flipAlternate) + one manual PlaceEdit(shift) on slice 5.
  2. Change rip from 0° to 15°. The pipeline regenerates.
  3. Cut produces a new slice list; provenance (contributing strips) is stable for interior slices.
  4. Preset re-expands against the new slice list — every odd slice gets flipped.
  5. PlaceEdit(shift) on slice 5 re-resolves: provenance matches → ok.
  6. The user sees the new mitered checker, the shift preserved, and a trace confirming which resolvers hit.

9.8 — Upstream change breaks a reference

  1. Same design as 9.7.
  2. User reduces pitch so Cut now produces 4 slices instead of 8. Slice 5 no longer exists.
  3. Pipeline regenerates:
    • Preset re-expands over 4 slices — 2 flips. ok.
    • PlaceEdit(shift) on slice 5: provenance has no match; sliceIdx 5 is out of range → error.
  4. The PlaceEdit's tile gets a red badge; timeline-panel row highlights red.
  5. User clicks the red badge. Options:
    • Reselect: pick a replacement slice in the join tile; status flips to ok.
    • Delete: remove the PlaceEdit entirely.

10. Error / warning taxonomy

Status Meaning Trigger examples User action
ok Reference resolved exactly. Default state; all inputs valid; provenance exact match. None.
warning Reference resolved approximately (fallback hit). Provenance set changed but sliceIdx still in range. User likely wants to review. Confirm, reselect, or delete.
error Reference cannot be resolved. Feature skipped. Target arrangeId gone (pass removed). sliceIdx out of range AND no provenance match. Spacer's anchor slice gone entirely. Reselect (from timeline panel) or delete the feature.
Deliberate non-goals. No auto-heal heuristics (nearest-centroid, nearest-species, etc.). No degrees of error beyond these three. No "multi-candidate" ambiguity state — if a resolver has multiple plausible matches, pick the first by a deterministic tiebreaker and mark it warning.

12. Open questions

  • Timeline panel docking — bottom drawer or right sidebar? Expanded or collapsed by default? Decision deferred to v2.13.
  • "Exclude from preset" UI — workflow 9.4 assumes a way for users to override a preset's effect on a single slice. Cleanest model: manual PlaceEdit(rotate 180) after the Preset cancels the preset's rotation on that target. Worth a UI affordance (right-click → "exclude from preset"). Deferred to v2.12 implementation.
  • Preset ordering vs. manual overrides. A manual PlaceEdit placed before a Preset vs. after means different things. UX probably wants "manual always wins" regardless of order. Spec needs a decision; pending v2.12.
  • Reference scope beyond slice and strip — e.g., a specific face or edge of a slice. Not needed for v2.1 scope; deferred until concrete use case appears.
  • Multi-pass narrow-column layout. At 3+ passes, each tile column gets narrow. Layout switch (2×N grid? pagination? zoom-to-tile?) deferred until the annoyance hits in practice.
  • Undo / redo. Not in v2 plan. State is an ordered list of features; history-stack is mechanically straightforward. Natural v2.18 follow-up.

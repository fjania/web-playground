Internal representation of every data structure in the pipeline

2. Input Types

StripPattern

The user edits a repeating unit of strips. The pipeline tiles it.

{
  unit: Strip[],        // the repeating sequence
  repeat: number,       // how many times to tile
  tail?: Strip[],       // optional strips appended after all repeats
}

Strip

{
  species: string,      // key into SPECIES lookup (e.g. 'maple', 'walnut')
  width: number,        // mm — the strip width as ripped on the table saw
}

Brick example with tail

{
  unit: [{ species: 'maple', width: 50 }, { species: 'walnut', width: 6 }],
  repeat: 4,
  tail: [{ species: 'maple', width: 50 }]   // → 9 strips: ■□■□■□■□■
}

3. Core State: WorkState

After each operation, the wood is in one of two states:

Panel (single piece)

{
  kind: 'panel',
  workpiece: Workpiece
}

Slices (multiple pieces)

{
  kind: 'slices',
  slices: Workpiece[],
  direction?: 'vertical'
}
Key rule: every operation takes a WorkState and returns a WorkState. Some operations require kind: 'panel' (crosscut, flatten, trim), others require kind: 'slices' (stack, rotate90, flip, shift, insert). Glueup works on both (initial from null, re-glue from slices).

4. Workpiece (3D-aware)

Every piece of wood — whether a panel or a single slice — is a Workpiece with 3D dimensions and a 2D face.

{
  width: number,        // mm — horizontal extent of the visible face
  height: number,       // mm — vertical extent of the visible face
  thickness: number,    // mm — depth into the board (the dimension you can't see)
  grainDir: string,     // 'long-grain' | 'end-grain'
  face: Face,           // the 2D renderable surface
  _cutAngle?: number,   // metadata: angle of the crosscut that created this slice
  _strips?: Strip[],    // metadata: original strip sequence for rotate90 to use
}

5. Face (2D geometry)

The renderable surface of any piece. Contains TWO kinds of geometry — rects and polys — which coexist on the same face.

{
  width: number,        // mm — bounding width
  height: number,       // mm — bounding height
  rects: Rect[],        // axis-aligned rectangles (0° cuts, simple patterns)
  polys: Poly[],        // arbitrary polygons (angled cuts, diagonal bands)
}

Rect

{
  x: number,    // left edge
  y: number,    // top edge
  w: number,    // width
  h: number,    // height
  species: string
}

Used for: 0° crosscut slices, initial glueup, checkerboard cells, brick cells

Poly

{
  points: [x, y][],  // vertices
  species: string
}

Used for: angled crosscut bands, chevron stripes, diagonal patterns

Known duality problem

Rects and polys are two separate lists on the same face. The renderer draws both. Some operations only transform rects (shiftAlternate), some only produce polys (angled crosscut). When you glue slices that mix rects and polys, both get offset and merged — but operations like shiftAlternate only process rects and leave polys untouched.

6. Snapshot

{
  op: Operation,        // deep clone of the operation that ran
  workState: WorkState   // deep clone of state AFTER this operation
}

The snapshots array is the full history. The pipeline cards show each snapshot. The result card shows snapshots[last].workState rendered directly (not via stage renderer).

7. Operations & What They Do to State

OperationInputOutputWhat changes
glueup null or slices panel Initial: strips → face rects. Re-glue: stacks slice faces along direction, merging all rects + polys with offsets.
crosscut panel slices Clips panel face into N pieces. At 0°: clips rects + polys to horizontal bands. At angle: creates parallelogram poly bands from _strips metadata.
stack slices slices Auto-rotates each slice 90° CW if height > width (longest edge horizontal). Sets direction: 'vertical'.
rotate90 slices slices Swaps face to end-grain view. face.height = stockThickness. thickness = old face.height. grainDir = 'end-grain'. Rebuilds face from _strips.

shiftAlternate slices slices Shifts odd slices by shift mm with wrap-around. Only processes rects.
insertStrips slices slices Interleaves new slices between existing ones. New slice face = full-width rect of specified species.
flatten panel panel Sets thickness = targetThickness. Face geometry unchanged.
trim panel panel Finds largest rectangle fully covered by content via scanline sampling. Clips + shifts all geometry.

8. Dimension Flow Through Operations

9. The Crosscut Problem (0° vs Angled)

0° Crosscut — clips actual face content

Slices the panel with horizontal lines. Each rect and poly on the face gets clipped to the band [yStart, yEnd] using geometric intersection (Sutherland-Hodgman for polys). The slice shows exactly what was on the panel in that band.

Angled Crosscut — creates NEW geometry from _strips

Does NOT clip the actual face. Instead, reads _strips metadata and generates fresh parallelogram band polys. The face content is thrown away and rebuilt.

This is the fundamental problem

A second angled crosscut on an already-assembled panel would need to clip the existing pattern (which could contain rects, polys, or both). Instead it reads _strips and creates fresh bands, losing the actual pattern. Two fix attempts broke stacking and were reverted.

10. The Rotate90 Face Swap

This is the conceptually trickiest operation. When you rotate a slice 90°, you're changing which face you see:

Why not just rotate the geometry? Because the long-grain face and end-grain face show completely different things. The long-grain face shows the flat side of strips (each strip is a tall rectangle spanning the panel length). The end-grain face shows the cross-section — each strip is a short band whose width = strip.width and height = stockThickness. These are different physical surfaces, so the geometry must be rebuilt.

11. Angled Cut Geometry Detail

For angle > 0, both crosscut and rotate90 produce parallelogram strip bands. The key parameter is skew.

// In crosscut (long-grain face):
skew = sliceThickness × tan(θ)
faceW = panelWidth / cos(θ) + skew

// In rotate90 (end-grain face):
skew = stockThickness × tan(θ)
faceW = panelWidth + skew
(crosscut) uW = strip.width (rotate90)

12. _strips Metadata — What It Is and Why It Exists

_strips is an array of { species, width } attached to each slice by crosscut. It records the original strip sequence of the panel that was cut.

Who writes it

crosscut — extracts strips from the panel face rects (sorted by x), attaches to each slice.

Who reads it

ConsumerHow it uses _strips
rotate90 Rebuilds the end-grain face from strip positions + skew. The actual face geometry from crosscut is discarded.
crosscut (angled) Uses _strips to generate parallelogram bands instead of clipping the face. Falls back to ctx.strips if _strips is empty.
_strips is a lossy shortcut

It assumes the panel was made from simple parallel strips. After a panel has been through flipAlternate + glueup (creating a checkerboard), _strips can't describe the face anymore. This is why angled crosscut on assembled panels doesn't work — the face has a complex pattern that can't be reconstructed from the original strip widths.

13. Stack Auto-Orientation

Stack has no parameters. It just ensures each slice's longest edge is horizontal by rotating 90° CW if needed.

// 90° CW rotation transform:
(x, y) → (oldHeight − y, x)

// Applied to rects:
{ x: oldH - r.y - r.h, y: r.x, w: r.h, h: r.w }

// Applied to poly points:
[x, y] → [oldH - y, x]

14. Flip & Shift Transforms

flipAlternate (horizontal mirror)

// Mirrors odd slices:
rect.x → face.width − x − w
poly point [x,y] → [face.width − x, y]

shiftAlternate (wrap-around offset)

// Shifts odd slices left by `shift` mm:
rect.x → rect.x − shift
// Rects that cross x=0 or x=width
// get SPLIT into two pieces.

15. Trim — Scanline Approach

Trim finds the largest axis-aligned rectangle fully covered by content. Used after chevron glueup to remove the jagged edges from angled cuts.

// Algorithm:
1. Convert all rects to polygon form (4 corners)
2. Sample 50 horizontal scan lines across face height
3. For each scan line, intersect with all polygon edges
4. Track the leftmost and rightmost x with content
5. xMin = max(all rowLefts), xMax = min(all rowRights)
6. Clip all geometry to [xMin, xMax] × [0, height]

17. Rendering Layer

drawLayout(svg, face, box)

The ONE rendering function. Takes a Face and a bounding box, scales to fit, draws all rects and polys as SVG elements.

// Scaling:
scale = min(box.w / face.width, box.h / face.height)

// Each rect:
<rect x={ox + r.x*scale} y={oy + r.y*scale} width={r.w*scale} height={r.h*scale} fill={SPECIES[r.species].color}/>

// Each poly:
<polygon points={p.points.map(([x,y]) => `${ox+x*scale},${oy+y*scale}`)} fill={SPECIES[p.species].color}/>

19. Known Issues & Limitations

1. Angled crosscut on assembled panels

Crosscut at angle > 0 ignores the actual face content and rebuilds from _strips metadata. This means a second crosscut on a re-glued panel (like cutting a checkerboard at 45°) produces plain strip bands instead of the actual checkerboard pattern cut diagonally. Two fix attempts (shear-clip-unshear) produced faces with incorrect bounding boxes that broke stacking.

2. shiftAlternate ignores polys

Only shifts rects. If a slice has poly geometry (from angled cuts), the polys stay in place while rects shift.

3. _strips assumes simple parallel strips

After operations that modify the pattern (flip, shift, re-glue), _strips no longer describes the face. rotate90 still reads _strips to build the end-grain face, which is correct for the first pass but would be wrong for a second rotate on an already-modified panel.

4. Trim uses sampling, not exact geometry

50 scan lines means the trim rectangle is approximate. For nearly-horizontal polygon edges, this could miss the tightest constraint. Generally good enough in practice.
